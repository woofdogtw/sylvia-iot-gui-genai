## Context

The shell's postbuild script (`inject-import-map.mjs`) currently:
1. Copies `vue.runtime.esm-browser.prod.js` → `dist/spa/assets/vue.js`
2. Injects an import map (`vue` + `quasar`) into `index.html`

The `quasar` entry in the import map points to `assets/quasar.js`, which is currently generated by Rollup `manualChunks`. That chunk only contains the Quasar exports that the shell itself imports — external plugins referencing any other component receive `undefined`. Additionally, Quasar CSS is tree-shaken at the component level by `@quasar/app-vite`, so only styles for shell-used components appear in the built CSS; plugin-used components render unstyled.

## Goals / Non-Goals

**Goals:**
- `assets/quasar.js` exports every Quasar component and plugin without requiring a shell rebuild.
- `assets/quasar.css` (full Quasar stylesheet) is automatically loaded so plugin components render correctly.
- No new npm dependencies; no CDN reliance.

**Non-Goals:**
- Minifying `vue.js` (it already ships as a production build).
- Expanding the set of Quasar plugins installed into the Vue app (out of scope).
- Supporting Quasar addon CSS (`quasar.addon.prod.css`) — not needed for standard components.

## Decisions

### Decision 1: Use `quasar/dist/quasar.client.js` as the source

`quasar.client.js` is the only ESM browser build Quasar ships in its npm package. All other candidates are rejected:

| Candidate | Format | Verdict |
|---|---|---|
| `quasar.client.js` | ESM (`export {}`) | ✅ chosen |
| `quasar.umd.prod.js` | UMD (factory wrapper) | ❌ not ESM — `import { X } from 'quasar'` cannot resolve named exports |
| `quasar.server.prod.js` | SSR / CJS | ❌ not for browser |
| CDN URL | ESM (external) | ❌ adds network dependency and CORS risk |

`quasar.client.js` imports from `"vue"` as a bare specifier, which the import map resolves to `/assets/vue.js` — preserving the singleton guarantee.

### Decision 2: Minify with esbuild (Node API)

`quasar.client.js` is unminified (~1 MB). esbuild is already present as a transitive dependency of Vite. Using its Node API (`import { transform } from 'esbuild'`) in `inject-import-map.mjs` reduces the output to ~507 KB without adding any new package or build step.

```js
import { transform } from 'esbuild'
const src = readFileSync(require.resolve('quasar/dist/quasar.client.js'), 'utf8')
const { code } = await transform(src, { minify: true })
writeFileSync('dist/spa/assets/quasar.js', code)
```

`inject-import-map.mjs` already uses `createRequire` for resolving vue; the same pattern applies here.

### Decision 3: Remove `manualChunks` for quasar from `quasar.config.js`

With quasar now provided externally as a copied file, the Rollup `manualChunks: { quasar: ['quasar'] }` config and its associated `chunkFileNames` override are no longer needed. The `external: ['vue']` setting is retained — it makes the shell bundle emit `import ... from 'vue'` statements that resolve via the import map.

Quasar itself does NOT need to be marked as `external` in Rollup because the shell still bundles quasar internally (for its own components). The import map is only used by external plugins. The shell's internal quasar imports are resolved at build time.

### Decision 4: Copy `quasar.prod.css` and inject `<link>` tag

`quasar/dist/quasar.prod.css` is the official pre-minified full Quasar stylesheet (~199 KB; gzip ~25 KB). It is copied to `dist/spa/assets/quasar.css` and injected into `index.html` as:

```html
<link rel="stylesheet" href="/assets/quasar.css">
```

Injection is done in `inject-import-map.mjs` alongside the existing import map injection. The `<link>` is placed immediately after the import map `<script>` tag (both inside `<head>`), before the shell's own CSS bundle, so the shell's styles take precedence when there is any overlap.

## Risks / Trade-offs

- **CSS duplication**: The shell's built CSS already contains Quasar styles for its own components. Adding the full `quasar.prod.css` duplicates ~50–100 KB of CSS rules. Duplicate CSS rules are harmless (last-write wins, same values). Gzip compression makes the actual transfer cost negligible.

- **esbuild API compatibility**: The `transform` API used here is the stable esbuild core API and has been unchanged since v0.8. It is only called in the postbuild script, not in the Vite pipeline itself, so a Vite upgrade that bumps esbuild would not break the interface.

- **`quasar.client.js` stability**: This is a first-party dist file and is stable within a major version. It will need re-evaluation on a Quasar v3 upgrade.

- **All 18 Quasar plugins installed**: Plugin code is already present in `quasar.client.js`, so there is no bundle-size penalty. Runtime initialization overhead is estimated at < 50 KB heap — negligible.

## Migration Plan

1. Run `npm run build` in `packages/shell` — the postbuild script produces the new `quasar.js` and `quasar.css` automatically.
2. Serve `dist/spa/` and verify that external plugins can import any Quasar component and render it with styling.
3. **Rollback**: revert `quasar.config.js` (restore `manualChunks`) and revert `inject-import-map.mjs` (restore `copyFileSync` from build output). No data or API changes to reverse.
